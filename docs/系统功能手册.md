# 数据分析 Agent 系统功能手册

> 本文档描述以 `langgraph dev` 启动后，系统的完整功能、工具体系、分层设计理由以及各 Agent 的提示词逻辑。

---

## 一、如何启动

```bash
# 在项目根目录执行
langgraph dev
```

启动后 LangGraph Studio 访问 `http://localhost:2024`，入口图为 `graph.py::data_agent`。

---

## 二、系统整体架构

```
用户请求
    │
    ▼
Supervisor Agent（主 Agent，create_react_agent）
    │  拥有 14 个工具：文件系统(4) + TODO(4) + 子Agent派发(3+1) + 语义层管理(1) + 会话管理(2)
    │
    ├─► delegate_to_sql_specialist
    │       └─ SubAgentMiddleware._run_sql_specialist()
    │             ├─ 语义层检索（向量 / token / 兜底）
    │             ├─ _generate_sql_with_llm（LLM + llm_skill 生成 SQL）
    │             └─ sql_inter（执行 SQL）
    │
    ├─► delegate_to_data_analyst
    │       └─ SubAgentMiddleware._run_generic_subagent()
    │             └─ 单轮 LLM 推理（python_inter / llm_skill）
    │
    └─► delegate_to_visualization_specialist
            └─ SubAgentMiddleware._run_generic_subagent()
                  └─ 单轮 LLM 推理（fig_inter / llm_skill）
```

## 三、工具全清单

### 3.1 文件系统工具（FilesystemMiddleware，4 个）

| 工具名 | 输入 | 作用 |
|--------|------|------|
| `ls` | `path?: str` | 列出 `/files/` 或 `/memories/` 目录 |

### 3.5 子 Agent 内部工具（不暴露给主 Agent，6 个）

| 工具名 | 所属 Agent | 输入字段 | 作用 |
|--------|-----------|----------|------|
| `sql_inter` | sql_specialist | `sql_query: str` | 执行 MySQL SELECT 查询，返回结果列表 |

| `fig_inter` | visualization_specialist | `py_code: str, fname: str` | 执行 matplotlib/seaborn 绘图代码并保存图片 |
| `table_metadata` | sql_specialist | `table_name: str` | 查询 INFORMATION_SCHEMA 获取表统计信息 |

## 四、工具分层设计

### 为什么要分层？

```
主 Agent 看到的工具（14 个）          子 Agent 内部工具（6 个）
━━━━━━━━━━━━━━━━━━━━━━━━━━━         ━━━━━━━━━━━━━━━━━━━━━━━━━
文件系统 + TODO + 派发工具              sql_inter / extract_data
抽象层：intent → delegate              python_inter / fig_inter
                                        llm_skill / table_metadata
                                        实现层：真实执行
```

**核心问题：把所有工具都给主 Agent 会导致什么？**

| 问题 | 影响 |
|------|------|
| 全量工具描述占满 context window | 每轮推理消耗 ~5000 tokens 的工具描述 |
| 主 Agent 直接选 `sql_inter` 而不经过语义层 | 跳过了表结构预取，SQL 准确率低 |
| 主 Agent 自己执行 SQL + 分析 + 绘图 | context 急速膨胀，超出 token 限制 |
| 无上下文隔离 | 上一个任务的 SQL 结果污染下一个分析任务的推理 |

**分层后的收益：**
- 主 Agent context 只含 14 个高抽象工具，节省约 70% 的工具描述 token
- sql_specialist 执行前自动经过语义层检索 → SQL 准确率显著提升
- 每个子 Agent 独立上下文，互不污染
- 子 Agent 内 LLM 调用不走 tool-call 消息链，避免 SiliconFlow 20015 错误

## 五、数据语义层（向量检索）

这是 `sql_specialist` 执行前的核心预处理阶段，**只在首次 SQL 任务时构建，后续全部走缓存**。

### 5.1 构建流程

```
首次调用 delegate_to_sql_specialist
        │
_build_semantic_layer()                   ← 仅执行一次
  ├─ 查全库表名（LIMIT 100）
  ├─ 每张表：
  │    ├─ 查列定义（column_name, data_type, column_comment，LIMIT 50）
  │    ├─ 取 3 行样例数据
  │    └─ 调 table_metadata 获取行数/大小/更新时间
  └─ embed_documents（BAAI/bge-m3）→ 每表一个向量
        │
存入 self._semantic_layer（进程内缓存）

后续调用
        │
_get_or_build_semantic_layer()
        └─ self._semantic_layer is not None → 直接返回缓存 ✓
```

### 5.2 向量描述文本格式

```
表名:device 字段:id name status type_id created_at 说明:主键 设备名称 状态 设备类型 创建时间 记录所有注册设备信息
```

中英文混合，`BAAI/bge-m3` 对中文友好，可正确理解"设备" ↔ `device` 这类映射。

### 5.3 检索优先级

```
task_prompt = "查询最近 7 天异常设备数量"
        │
① 向量检索：embed(task_prompt) → cosine 相似度排序 → top-4
        │ 失败（embedding API 不通）
② 英文 token 匹配：从 task_prompt 提取英文单词，精确匹配表名
        │ 无命中
③ 兜底：取前 4 张表（按字母排序）
```

返回的 `schema_context` 示例：
```
数据库所有表(8张): ['alarm', 'device', 'orders', ...]
[检索方式: 向量检索 top4: device(0.823), alarm(0.761), status(0.642), logs(0.511)]
表 device 字段: id(int), name(varchar), status(tinyint), created_at(datetime)
表 device 样例: [{"id":1,"name":"传感器A","status":0,...}, ...]
```

### 5.4 手动刷新

表结构变更后，主 Agent 可调用：
```
rebuild_data_semantic_layer()
→ ✅ 数据语义层已重建，共 8 张表: ['alarm', 'device', ...]
```

## 六、SQL 执行格式链（重点验证）

### 6.1 完整格式链

```
_generate_sql_with_llm()
        │
llm_skill(prompt=..., json_mode=True, output_schema={"sql_query": str})
        │
LLM 返回：{"sql_query": "SELECT COUNT(*) FROM device WHERE status=0"}
        │
_extract_sql_from_text()（4 种场景依次尝试）
  场景1：json.loads(text).get("sql_query")          → 纯 JSON 返回         ✓
  场景2：```json{"sql_query":"..."}``` 代码块        → Markdown JSON 包裹   ✓
  场景3：```sql SELECT ... ``` 代码块                → SQL 代码块           ✓
  场景4：裸 SELECT...(分号可选)                      → 纯 SQL 文本          ✓
        │
sql_query = "SELECT COUNT(*) FROM device WHERE status=0"
        │
_invoke_tool_by_name(tool_map, "sql_inter", sql_query=sql_query)
  → tool_obj.invoke({"sql_query": "SELECT COUNT(*) FROM device WHERE status=0"})
        │
SQLQueryInput(sql_query="SELECT COUNT(*) ...") ← Pydantic 验证     ✓
        │
SQLQueryTool._execute(sql_query=...)
  → _is_safe_query()：SELECT 通过，禁用 INSERT/UPDATE/DELETE/DROP/CREATE/ALTER/TRUNCATE
  → DatabaseManager.execute_query(sql_query)
  → ToolResult(success=True, data=[...], message="查询成功，共返回 N 条记录")
        │
BaseCustomTool._run() → to_tool_message_content() → str
        │
sql_result_text = str(sql_result)   ← 最终为字符串
```

### 6.2 错误处理路径

```
_generate_sql_with_llm 返回 None（llm_skill 不可用 / 解析失败）
        │
_select_sql_for_task(task_prompt)   ← 关键词规则兜底
  "多少" + "表" → COUNT(*)
  "表结构" / "字段" → information_schema.columns
  "所有表" / "表名" → information_schema.tables

sql_inter 返回包含 "执行失败"
        │
返回：SQL 查询执行失败:\n- SQL: ...\n- 错误: ...\n- 建议: 检查数据库连接配置
```

### 6.3 已知注意事项

| 项目 | 说明 |
|------|------|
| 仅支持 SELECT | `_is_safe_query` 硬性拦截写操作，确保只读 |
| 结果上限 | 超过 1000 条自动截断，返回前 1000 条 |
| 分号处理 | `_extract_sql_from_text` 的裸 SQL 场景4支持无分号结尾（`?` 可选匹配）|
| 反引号表名 | `_collect_table_context` 样例查询使用 `` SELECT * FROM `{table}` `` 防止表名含关键字 |

## 七、各 Agent 提示词说明

### 7.1 Supervisor Agent

**文件**：`agents/supervisor.py::SUPERVISOR_PROMPT`

**核心设计**：强制工具调用规则放在提示词**最顶部**，确保优先级最高。

```
【强制规则 - 最高优先级】
你必须通过调用工具来完成任务，禁止用文字描述"我可以…"
判断规则：
- 数据库/查询/统计 → 立即调用 delegate_to_sql_specialist
- 数据分析/统计计算 → 立即调用 delegate_to_data_analyst
- 图表/可视化       → 立即调用 delegate_to_visualization_specialist
❌ 错误："我可以通过执行 SHOW TABLES... 如果你提供连接信息..."
✅ 正确：[调用 delegate_to_sql_specialist，task="查询所有表"]
```

**后续内容**：
- 角色：协调多子 Agent 的 Supervisor
- 子 Agent 说明（名称、职责、输出规范）
- 文件系统路径规范（`/files/` 临时，`/memories/` 持久）
- 任务规划示例（create_todo + delegate 流程）

---

### 7.2 SQL Specialist 提示词

**文件**：`agents/subagents/sql_agent.py::SQL_SUBAGENT.system_prompt`

> **注意**：该提示词现在不直接驱动 LLM 推理（`_run_sql_specialist` 是直接执行管道），但通过 `_run_generic_subagent` 路径仍会使用。

**核心内容**：

```
1. 理解查询需求 - 分析意图，识别表/字段/条件
2. 生成优化 SQL - 高效写法，处理 JOIN/子查询/聚合
3. 执行并验证 - 执行 sql_inter，验证结果正确性
4. 保存结果 - 写入 /files/sql_result_{task_id}.json

工具选择策略：
- 纯文本总结/字段释义 → 优先 llm_skill，不要构造无意义 SQL
- 需要真实数据库读取 → 使用 sql_inter
- 需要 DataFrame → 使用 extract_data
```

---

### 7.3 Data Analyst 提示词

**文件**：`agents/subagents/data_analysis_agent.py::DATA_ANALYSIS_SUBAGENT.system_prompt`

**核心内容**：

```
1. 数据读取    - 从 /files/ 读取 JSON/CSV/Parquet
2. 数据清洗    - 缺失值、异常值、类型转换
3. 统计分析    - 描述性统计、聚合分析、相关性/趋势分析
4. 生成洞察    - 识别模式、提供业务含义
5. 保存结果    - 写入 /files/analysis_{task_id}.json

工具选择策略：
- 总结/改写/分类/结构化提取 → 优先 llm_skill
- 需要计算/pandas 处理       → python_inter
- 不要把纯文本任务误判为 Python 分析任务
```

---

### 7.4 Visualization Specialist 提示词

**文件**：`agents/subagents/visualization_agent.py::VISUALIZATION_SUBAGENT.system_prompt`

**核心内容**：

```
1. 数据读取 - 从 /files/ 读取分析结果，理解数据结构
2. 图表设计 - 选择合适类型（柱状/折线/饼图/散点/热力/箱线）
3. 图表创建 - matplotlib/seaborn/plotly
4. 样式要求 - 专业配色、清晰标签、fig.tight_layout()
5. 保存输出 - images/chart_{task_id}.png

关键约束：
- 坐标轴/标题/图例必须使用英文（避免中文字体缺失乱码）
- 必须用 fig = plt.figure() 或 fig, ax = plt.subplots()
- 禁止 plt.show()

工具选择策略：
- 总结图表结论/改写说明 → llm_skill
- 实际绘图              → fig_inter
```

## 八、高级特性说明

### 8.1 SiliconFlow 兼容性

系统专为 SiliconFlow（DeepSeek-V3）设计，规避了以下限制：

- **20015 错误**：SiliconFlow 拒绝包含空 `content` 的 AIMessage 和多轮 ToolMessage 链。解决方案：子 Agent 完全不走 LLM tool-call 循环，使用直接执行管道。
- **消息格式**：子 Agent 内部 LLM 调用只用 `[SystemMessage, HumanMessage]` 单轮结构。

### 8.2 recursion_limit

`invoke()` 配置了 `"recursion_limit": 8`，防止主 Agent 工具调用陷入死循环。

### 8.3 文件持久化

`save_result_to` 参数触发 `_persist_execution_result()`，通过 `CompositeBackend.write_file()` 写入 `StateBackend._files` 字典，在同一会话内可通过 `read_file` 读回。**注意**：StateBackend 是内存存储，进程重启后清空。
